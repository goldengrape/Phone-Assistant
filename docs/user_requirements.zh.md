# 用户需求

## 1. 产品概述

AI Phone Assistant 是一个实时语音通话监督界面。

它面向一位人类操作员，目标是让 AI 模型以电话式语音对话的方式与对端沟通，而操作员可以：

- 监控通话
- 观察识别到的语音内容
- 通过静默文本引导 AI
- 调整 prompt 与语言设置

## 2. 核心用户

### 监督者（Supervisor）

监督者是本应用的主要用户。

监督者需要能够：

- 选择厂商/模型
- 配置 API 凭证
- 设置通话目标（call purpose / system instructions）
- 选择 AI 语音语言
- 在不影响 AI 语音语言的情况下独立选择界面语言
- 使用 Gemini 时选择一个 voice 预设
- 启动与结束实时会话
- 监控语音输入/输出电平
- 阅读转写时间线
- 在通话中发送静默引导指令
- 在会话结束后导出日志

## 3. 功能需求（FR）

- `FR-01`：系统必须提供基于浏览器的监督 UI。
- `FR-02`：系统必须支持 Gemini 与 Qwen 两个可选实时 provider。
- `FR-03`：系统必须允许在连接前编辑主 system instruction / call purpose。
- `FR-04`：系统必须为便利性将 API key 保存在浏览器本地。
- `FR-05`：系统必须采集麦克风音频并实时流式发送到当前 provider。
- `FR-06`：系统必须实时播放模型音频输出。
- `FR-07`：系统必须在 UI 中呈现识别到的用户语音与助手语音内容。
- `FR-08`：系统必须维护一条转写时间线，包含用户、助手、监督者与系统消息。
- `FR-09`：系统必须允许监督者在会话中发送静默指令。
- `FR-09a`：静默指令输入应在查看转写历史与实时监督时保持快速可达。
- `FR-10`：系统必须支持导出转写/日志。
- `FR-11`：UI 必须同时适用于桌面宽屏与窄窗口。
- `FR-12`：UI 必须适配系统浅色/深色偏好。
- `FR-13`：UI 至少支持中/日/韩/英/法/西。
- `FR-14`：设置区必须清晰区分界面语言与 AI 语音语言。
- `FR-15`：系统必须允许从支持的预置 voice 中选择 Gemini voice。
- `FR-16`：高级的 session context 诊断不应主导默认监督流程，应默认隐藏或可折叠。

## 4. 非功能需求（NFR）

- `NFR-01`：会话启动流程必须足够简单，非开发者也能用。
- `NFR-02`：provider 特定复杂性应隐藏在稳定的 UI 之下。
- `NFR-03`：监控信号必须清晰，让监督者能判断音频是否在进入/输出。
- `NFR-03a`：监控面板在转写区已展示信息时应避免重复展示文本内容。
- `NFR-04`：系统应尽量减少不必要的基础设施，核心本地体验不强制依赖后端。
- `NFR-05`：系统应能容忍不同 provider 的输出采样率差异。
- `NFR-06`：核心交互控件，尤其是静默干预，应尽量减少滚动成本。

## 5. 当前范围边界

当前范围包含：

- 浏览器运行时
- 实时音频通话监督
- prompt 引导
- 多语言 UI
- 主题适配

当前不承诺的产品功能：

- 云端持久化存储
- 多人协作
- 重型 skill engine
- CRM 集成
- 通话分析后端
