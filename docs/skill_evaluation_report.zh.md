# Skill 评估报告

## 1. 问题

AI Phone Assistant 是否应该加入 `AI skill` 层？

这里的 skill 指一种可复用的行为包，用于让模型在特定通话场景中更稳定、更一致地表现，例如：

- 谈判
- 客服支持
- 线索筛选
- 预约与改期
- 升级与转接处理

## 2. 当前产品语境

当前产品是实时语音监督工具，而不是慢节奏的聊天 agent。

这会显著改变取舍：

- 延迟很重要
- prompt 体积很重要
- 语音场景比纯文本更在意语气一致性
- 冲突指令在语音输出中更容易被察觉

产品当前已经有一个轻量的控制面：

- 可编辑的 system instruction / call purpose

因此关键不是“能不能加 prompt”，而是“以什么形式加 skill 才适合实时语音产品”。

## 3. 加入 Skill 的收益

### 3.1 更好的复用

电话场景往往高度重复。skill 可以把最佳策略打包一次复用多次。

例子：

- 价格谈判
- 预约改期
- 安抚与降级
- 信息收集
- 销售跟进

### 3.2 更稳定的输出

没有 skill 时，操作员往往每次重写 call purpose。

这容易导致：

- 文案不一致
- 输出质量波动
- 每次通话前准备成本变高

合理的 skill preset 能提升行为可预测性。

### 3.3 更好的上手体验

新用户通常不会写“适合实时电话”的提示词。

skill 可以提供更靠谱的起点，降低门槛。

### 3.4 更容易评估迭代

有命名的 skill preset 后，更容易做对比与迭代：

- 哪种开场更有效
- 哪种谈判风格过于强势
- 哪种多语 framing 更自然

## 4. 加入 Skill 的风险

### 4.1 实时成本

重型 skill 系统通常会增加 prompt 长度与编排复杂度。

在实时语音应用中，这会伤害：

- 启动速度
- 响应延迟
- token 成本
- 稳定性

### 4.2 指令冲突

应用已有多层引导：

- 基础电话策略
- 可编辑 call purpose
- 输出语言约束
- 通话中监督者静默指令

skill 若过强，可能与以下内容打架：

- 当前通话目标
- 输出语言规则
- 临时静默指令

### 4.3 维护成本

重型 skill 框架不仅是 prompt 文本，还往往需要：

- 版本管理
- 测试
- 多语言调优
- 多模型调优
- UI 管理

这对仍在打磨实时音频链路的产品来说，负担过大。

### 4.4 语音模式的失败形态

聊天里能接受的表达，在语音里可能会显得不自然。

过于显式或“操控感强”的谈判策略可能听起来：

- 机械
- 过度脚本化
- 情绪不对
- 令人警惕

## 5. 评估结论

项目大概率应该加入 skill，但不应在当前阶段做成重型框架。

最佳结论：

- `是`：做轻量 skill preset/profile
- `否`：当前阶段不做重型 skill engine

## 6. 推荐产品路线

### Phase 1：轻量 Skill Profiles

把 skill 作为结构化 prompt preset，叠加在现有 system instruction 上。

每个 skill 应该足够短且明确，例如包含：

- 目标
- 语气
- 禁止行为
- 3 到 5 条策略
- 开场风格
- 收尾风格

这样能与现有架构很好组合：

- `callPurpose`
- 输出语言约束
- 静默指令

### Phase 2：小步扩展

只有当第一版 skill preset 证明有价值，才考虑：

- skill 参数化
- 每个 skill 的评估指标
- 可导入的 skill 库
- 通话类型模板

### Phase 3：避免过早上重系统

不要一开始就上：

- RAG 检索式 skill
- 多 skill 规划链
- 复杂 tool graph
- 大型规则引擎

这些能力很强，但对当前实时约束与产品成熟度来说过重。

## 7. 第一批推荐 Skill

若近期要加 skill，优先建议：

- `Negotiation`（谈判）
  - 目标：自然地争取更好的价格/条款
- `Sales Follow-up`（销售跟进）
  - 目标：推进下一步，处理异议
- `Customer Support / De-escalation`（客服安抚/降级）
  - 目标：稳定情绪，收集事实，维护信任
- `Information Collection`（信息收集）
  - 目标：高效收集结构化信息
- `Scheduling / Rescheduling`（预约/改期）
  - 目标：确认时间窗口并顺利收尾

## 8. 与架构的匹配方式

从当前代码基线看，skill 最合适的位置是：

- prompt 组装之上的一层轻量配置

而不是放在：

- 音频模块
- 播放逻辑
- 传输协议代码

因此未来实现更可能落在：

- store 配置
- prompt 组合
- UI preset 选择

## 9. 最终建议

当前阶段应以“轻量 preset”形式加入 `AI skill`。

这样能拿到主要收益：

- 复用
- 一致性
- 上手体验
- 场景调优

同时避免重型 skill 平台带来的延迟、复杂度与维护负担。
