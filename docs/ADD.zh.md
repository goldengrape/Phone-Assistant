# 公理化设计笔记

## 1. 设计意图

本文从公理化设计（Axiomatic Design）的视角描述当前 Web 版本实现。

核心产品目标是：

- 让人类监督者以尽可能低的操作摩擦，控制并监控一次由 AI 主导的实时电话式对话

## 2. 顶层功能需求

- `FR0`：提供一个实时语音通话监督界面，能够听、说、监控，并以“静默方式”引导 AI 电话助手。

- `DP0`：一个基于浏览器的 React 应用，包含本地音频 I/O、厂商适配器、转写监控，以及可编辑的 Prompt 控制。

## 3. 主要功能分解

- `FR1`：允许监督者配置模型、API Key、通话目标、界面语言和 AI 输出语言。
  - `DP1`：由 Zustand + local storage 持久化支撑的设置面板与 Prompt 面板。

- `FR2`：将麦克风音频实时流式发送到当前选中的厂商。
  - `DP2`：`AudioCapture` + 厂商特定的音频上行接口。

- `FR3`：以足够平滑的方式播放模型返回的实时语音。
  - `DP3`：`AudioPlayback`，包含 PCM 转换与音频分片调度播放。

- `FR4`：让监督者看到通话状态与识别到的语音内容。
  - `DP4`：转写时间线、输入/输出监控卡、转写预览回调与系统事件消息。

- `FR5`：允许监督者在通话中静默重定向模型行为。
  - `DP5`：静默指令输入框，映射到厂商特定的文本注入方式。

- `FR6`：支持多家厂商而不把厂商细节渗透到 UI 层。
  - `DP6`：`AIClient` 抽象层 + `GeminiLiveClient` / `QwenLiveClient` 适配器。

- `FR7`：保证在不同设备/窗口尺寸与不同语言环境下仍可用。
  - `DP7`：响应式 CSS、主题 token、以及本地化 UI 文案。

## 4. 独立性考虑

当前设计不是完全“无耦合”，但主要需求已经做到相对清晰的分离：

- UI 配置基本与传输逻辑隔离
- 音频采集与厂商传输细节隔离
- 播放与转写逻辑隔离
- 厂商差异集中在适配器类中
- 本地化通过翻译层与视图逻辑隔离

仍然存在的强耦合是合理且可接受的：

- 实时连接状态同时影响 UI 控件与音频生命周期
- Prompt 组装依赖“厂商连接配置 + 用户配置”

## 5. 信息量最小化

当前实现相比早期设想更轻量：

- 不强制引入后端服务
- 不要求桌面桥接程序
- 前端主路径不依赖 Python 运行时
- 不引入重量级编排层来连接 UI 与厂商

这降低了部署与调试复杂度，符合信息公理。

## 6. 当前设计取舍

一个有意保留的“非最小化”选择是 Gemini 集成：

- 项目使用官方 `@google/genai` SDK，而不是维护一套低层协议客户端

这会增加前端 bundle 体积，但能降低协议漂移风险，并与官方支持的实时接口保持一致。

## 7. 对未来 Skill 的含义

如果未来加入 `AI skill` 支持，也应保持相同的设计原则：

- 除非能解锁明确价值，否则不要引入重量级 skill 引擎
- 优先使用轻量、结构化的 skill profile/preset，并与现有 prompt 组装方式可组合
