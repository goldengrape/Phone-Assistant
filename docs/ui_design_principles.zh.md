# UI 设计原则

## 1. 目的

本文为 AI Phone Assistant 定义 UI 设计原则，用于保证后续界面改动与“实时监督工作流”保持一致。

本产品不是通用聊天机器人，而是用于监控与引导实时 AI 语音通话的操作台。

## 2. UI 的核心目标

UI 应帮助监督者快速完成三件事：

- 理解此刻正在发生什么
- 以最小摩擦改变 AI 行为
- 在实时通话中保持上下文与方向感，同时避免视觉噪音

## 3. 信息层级

UI 应按以下优先级呈现信息：

1. 连接状态与当前模型
2. 实时转写与系统事件
3. 快速静默干预（whisper）
4. 核心配置项
5. 高级诊断信息

含义：

- 转写与静默干预属于主路径
- 高级诊断属于次路径，默认应折叠

## 4. 控件布局

### 4.1 Whisper 优先

静默指令栏是核心监督控件，而不是次要操作。

因此它应当：

- 固定或粘附在视口底部附近
- 在监督者滚动查看历史转写时仍可用
- 支持高频、重复输入

### 4.2 设置与监控分离

设置项应与转写流分离，并以“配置”而不是“对话内容”的视觉语义呈现。

配置区域应对齐、易扫读。

## 5. 语言设定

UI 必须明确区分两个“语言”概念：

- 界面语言：软件 UI 使用的语言
- AI 语音语言：模型语音输出应使用的语言

这两者在标签与布局中不得混淆。

## 6. 监控面板

Voice input / output 卡片的目标是回答：

- 是否有音频活动？
- 系统当前在听还是在说？

如果主转写区已经呈现转写内容，监控卡不应重复大段文本。

优先设计：

- 更小巧的卡片
- 清晰标签
- 电平条
- 简单状态徽章

## 7. 转写区

转写区是通话历史的主记录。

应包含：

- 用户最终转写
- 助手最终转写
- 监督者静默指令消息
- 系统事件
- 需要时的实时预览文本

转写区首先要“可读”，其次再追求装饰性。

## 8. 视觉对比度

浅色与深色主题下的可读性必须被保证。

重点包括：

- 转写气泡
- 时间戳与角色标签
- muted 辅助文字
- 带 tinted 背景的卡片

柔和的视觉风格不能牺牲核心内容的可读性。

## 9. 渐进披露

高级信息应当“可用但不打扰”。

例如：

- session context
- 模型诊断信息
- 传输细节

除非信息可立即行动，否则应放在折叠区域中。

## 10. 响应式

布局需要同时适配宽屏监督与窄窗口。

在小宽度下：

- 控件应自然堆叠
- 转写仍需可读
- 底部静默指令栏必须可用
- 设置区应自适应重排且不产生难看的错位

## 11. 需要保留的偏好

遇到取舍时，优先：

- 清晰大于密度
- 速度大于装饰
- 对齐大于花哨的不对称
- 监督工作流大于“仪表盘感”
- 可折叠的复杂性大于默认常驻的诊断信息

## 12. 实用检查清单

在加入任何新的 UI 区块前，先问：

- 是否让操作员更快行动？
- 是否必须常驻显示？
- 是否在重复已有信息？
- 是否在浅色/深色主题下都清晰？

如果第 2 或第 3 个问题答案不理想，这个区块更可能应该被折叠、简化或迁移位置。
